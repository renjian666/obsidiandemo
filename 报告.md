### 三、 实验内容
本实验旨在设计并实现一个基于 FCFS（先来先服务） 算法的进程调度可视化程序。实验要求能够直观展示进程在不同状态（ 就绪、执行、阻塞、完成 ）之间的转换过程，并提供交互手段（如添加进程、手动阻塞、解除阻塞）来模拟真实的操作系统调度环境。

### 四、 实验原理（理论）
1. FCFS 调度算法 ：这是最简单的调度算法。它按照进程到达的先后顺序进行调度。当 CPU 空闲时，算法从就绪队列的头部取出一个进程并分配 CPU。该算法属于 非抢占式 调度，即一旦进程获得 CPU，就会一直运行直到完成或发生阻塞。
2. 进程状态模型 ：
   - 就绪 (Ready) ：进程已具备运行条件，正在排队等待分配 CPU。
   - 执行 (Running) ：进程获得 CPU，正在执行其任务。
   - 阻塞 (Blocked) ：进程因等待某些事件（如 I/O）而暂时无法运行。
   - 完成 (Finished) ：进程执行结束，释放资源。
3. 状态转换逻辑 ：
   - 就绪 -> 执行 ：调度程序选择队列首进程。
   - 执行 -> 完成 ：运行时间倒计时至零。
   - 执行 -> 阻塞 ：模拟 I/O 请求或其他等待事件。
   - 阻塞 -> 就绪 ：等待事件结束，进程重新进入就绪队列末尾。
### 五、 实验涉及仪器设备和材料清单
- 操作系统 ：Windows 11
- 开发语言 ：Java 21
- 图形界面库 ：JavaFX
- 构建工具 ：Maven
- 开发工具 ：Trae IDE / IntelliJ IDEA
### 六、 实验步骤及代码分析
1. 构建进程模型 ( Process.java ) ：
   定义进程的基本属性，如 id 、 remainingTime （剩余时间）和 state （状态枚举）。使用枚举类型明确区分进程的四个生命周期阶段。
2. 设计可视化界面 ( hello-view.fxml ) ：
   采用 VBox 和 HBox 嵌套布局。顶部为标题，中部左侧为“执行区”，中部右侧上下排列“就绪队列”和“阻塞队列”的容器。底部设置控制按钮和速度滑块。
3. 核心调度逻辑实现 ( HelloController.java ) ：
   - 时间片模拟 ：利用 JavaFX 的 Timeline 类，以 1 秒为周期触发 step() 方法，模拟时钟中断。
   - 调度策略 ：在 step() 中判断 runningProcess 是否为空。若为空且就绪队列不空，则执行 scheduleNext() 。
   - 动画实现 ：通过 TranslateTransition 计算进程节点在 Pane 之间的坐标差，实现平滑的物理位移动画，而非简单的“闪现”切换。
4. 交互控制 ：
   实现 onAddProcessClick （随机生成进程）、 onBlockProcessClick （强制当前进程让出 CPU 进入阻塞区）等方法，模拟复杂的系统行为。
### 七、 实验结果及分析
- 可视化效果 ：运行程序后，通过点击“添加进程”，可以看到代表进程的色块以缩放动画进入就绪队列。点击“开始”后，色块平滑移动至执行框，剩余时间开始递减。
- 算法验证 ：观察发现，无论进程的运行时间长短，系统严格按照它们进入就绪队列的先后顺序进行处理，符合 FCFS 算法的特性。
- 状态转换验证 ：手动点击“阻塞”后，进程立即停止执行并移动至阻塞区，CPU 转而处理就绪队列中的下一个进程。解除阻塞后，该进程回到就绪队列末尾，体现了 FCFS 算法中“重新排队”的规则。
- 性能观察 ：FCFS 算法虽然实现简单且公平，但存在“护航效应”（Convoy Effect），即长进程在前面时，会导致大量短进程在后面长时间等待。
### 八、 实验总结及心得体会
1. 对 FCFS 的深入理解 ：通过可视化模拟，我直观地感受到了 FCFS 的局限性。在多任务环境下，长任务对系统的响应时间有显著影响。
2. 可视化在教学中的价值 ：相比于枯燥的文字定义，动态的队列移动和状态切换能让人更迅速地理解操作系统底层调度的精髓。
3. JavaFX 动画编程体会 ：在实现进程位移动画时，难点在于跨容器的坐标转换。通过 localToScene 获取绝对坐标并计算偏移量，我掌握了处理复杂 UI 平滑过渡的技巧。
4. 系统设计思维 ：本实验不仅是算法的实现，更是对“模型-视图-控制器”（MVC）架构的一次实践。将进程的逻辑状态与 UI 的表现形式解耦，使得程序具备更好的扩展性，例如未来可以轻松切换到 RR（时间片轮转）或 Priority（优先级）调度算法。
